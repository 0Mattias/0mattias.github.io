Cisco Modeling Labs on Google Cloud

CML 2.10 on a Google Cloud VM for CCNP study: the nested virtualization flag, a Tailscale-only perimeter, and the two Docker bugs between install and the first booted lab.

July 6, 2026 · 5 min read

The CCNA is done and the CCNP is next, and CCNP labs want more routers than a bench can hold. The sanctioned way to get them is Cisco Modeling Labs: real Cisco network operating systems booted as virtual machines, wired into topologies in a browser. The catch is that CML wants a dedicated x86 host with virtualization extensions and a pile of RAM, and the hardware by my desk is three small single-board computers that already have jobs.

So the lab went to Google Cloud. This is the build log.

The instance

An n2-standard-8 in us-central1: eight vCPUs, 32 GB of RAM, a 100 GB disk, stock Ubuntu 24.04 Server. Enough for the multi-router topologies the CCNP material asks for, with room left over for the Alpine hosts and unmanaged switches that fill out a lab.

The one thing a stock instance lacks is nested virtualization. CML boots its nodes under KVM, so the VM has to pass the virtualization extensions through, and GCP keeps that flag out of the console UI. The dance: stop the instance, gcloud compute instances export, add advancedMachineFeatures.enableNestedVirtualization: true to the exported YAML, update-from-file, start it again. After that /dev/kvm exists inside the VM and CML has what it needs.

The install

Cisco ships CML as a .pkg that is a tarball wearing a different extension. Inside sit Debian packages: the cml2 controller, the PaTTY console proxy, the IOL tools. Extract them, apt install alongside network-manager, copy the reference platform images into /var/lib/libvirt/images, and the controller is on the box. A CML Personal license goes in later through the web UI as a Smart Licensing token.

Doing this headless on a cloud image has two wrinkles. The initial-setup service expects someone at a console TTY, so instead you hand it /etc/virl2-base-config.yml with interactive: false and the admin accounts predefined, strip the TTY directives from the service unit, and let it configure unattended. And CML’s network helper likes to switch netplan’s renderer to NetworkManager, which on Google Compute Engine kills runtime networking the instant it applies. I can report that a GCP instance reset is quick. The same change applies cleanly at boot; it just cannot be done live.

One more thing the package does without asking: it moves the system sshd to port 1122. Worth knowing before you conclude your VM is gone.

Nothing public

The VM joined the tailnet before CML went on, and every path in has stayed there: the web UI is HTTPS on the tailnet only, admin access is Tailscale SSH, node consoles run in the browser. There is no public 443. The default allow-SSH firewall rule is deleted, and break-glass is a single rule admitting Google’s Identity-Aware Proxy range, so a dead tailscaled still leaves a way in through an IAP tunnel.

A 2026 footnote: Google no longer grants the default service account Editor on new projects, so the VM cannot touch its own infrastructure. Every gcloud change runs from the laptop instead. That is the right default, and for a box whose job is booting lab routers, nothing of value is lost.

Two Docker bugs

CML boots two kinds of node. The KVM kind (IOSv, the Catalyst 8000v, qcow2 images) worked on the first try. The Docker kind (IOL, IOS XE compiled to run as a Linux process) did not: every IOL node died with Delayed docker node start failed, and the shim log said No such image, naming a SHA the Docker daemon insisted it had never seen.

Cause one: current docker-ce defaults to the containerd image store, and the containerd store assigns a loaded image a different ID than the classic graph driver would. CML’s image definitions record the classic ID and look the image up by it. Same tarball, same bytes, different name, no match. The fix is one line in /etc/docker/daemon.json, {"features": {"containerd-snapshotter": false}}, then a restart of Docker and the shim. Images load under the expected ID and every IOL node reaches BOOTED.

Cause two was mine. The reference images came over as tar piped through ssh from macOS, and macOS tar shipped 51 AppleDouble ._* files along with them. The shim then tried to parse the ._ copies of the image definition YAMLs as definitions and refused the control characters inside. Delete them, restart, done. The lasting rule: nothing leaves a Mac by tar without COPYFILE_DISABLE=1, and rsync exists.

The 40 milliseconds

Consoles felt slow, so I went hunting for the misconfiguration, and there was none to find. tailscale ping showed the WireGuard tunnel already direct, no relay, steady between 38 and 44 ms. That number is geography: us-central1 is in Iowa, I am in Boston, and 40 ms is roughly what the distance costs. The real fixes are moving the VM to an East Coast region someday, or accepting 40 ms of typing lag in a router console; the second turns out to be easy to live with. A larger instance would change nothing here. More vCPUs buy bigger topologies, not shorter round trips.

Running it

Seventeen platform images are registered, IOL and IOSv in router and switch flavors for the core work, plus CSR1000v and Catalyst 8000v, an ASAv, an NX-OS switch, a wireless AP and client, and Alpine and Ubuntu hosts. I skipped the Catalyst 9000v on purpose (18 GB of RAM per node has no place on a 32 GB box) and the service-provider images, which belong to a track I am not on.

The instance is stopped whenever I am not in a lab. The homelab’s Uptime Kuma keeps a monitor on the web UI anyway, which sounds backwards for a machine that is supposed to be down; in practice it is a did-I-leave-it-running light.

Nothing here is exotic once it runs. The instance starts, the topology boots, and the CCNP plan has a bench that powers off between sessions.
