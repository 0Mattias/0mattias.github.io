Building bettermemory.

Persistent memory for AI coding agents, retrieved on demand rather than pre-loaded. Notes on the design, and everything added since 2.0.

May 18, 2026 Updated June 11, 2026 9 min read

bettermemory is a local MCP server (and a Claude Code plugin). Memory lives as plain markdown on disk and gets exposed through MCP tools the model calls when it actually needs context. There is no automatic injection per turn. When the model does pull a memory in, the contract is to say so in the reply.

The failure mode

The shape most memory features ship with is the same: stored facts get pre-loaded into every conversation. After a few weeks of real use, two patterns show up.

The first is bleed-through. Memory pulled in for an unrelated topic still tints tone, framing, and assumptions. The model is doing what you would expect when irrelevant context lands in the prompt. The bug is the design choice that puts it there.

The second is silent staleness. A fact stored in March still sits in the prompt in November, after the file paths moved or the project changed direction. The model treats every memory as fresh.

False negatives are recoverable. The model forgets, you tell it again. False positives cascade: the model anchors on a stale fact, builds reasoning on top, and you don’t see the moment the conversation went off.

Opt-in retrieval

bettermemory exposes memory as a set of MCP tools the model calls when it needs them. At session start, memory_scope_overview returns counts per scope without bodies. A total of zero means nothing to fetch. Otherwise memory_search runs only when a request is ambiguous in a way stored context could resolve.

“Walk me through pandas from zero to hero” hits the search tool, because that phrasing is the kind a learning-style preference might shape. “How do I write a Python list comprehension” does not. When the model does retrieve and use a memory, it has to say so in the reply: “Using your stored preference for code-driven tutorials…”

Three staleness signals, one verdict

Memory is a snapshot. It does not refresh itself. Every retrieval carries three structured signals folded into a single staleness_verdict ∈ {fresh, spot_check_recommended, spot_check_required}.

Calendar. Each memory has a last_verified_at timestamp against a configurable threshold (default 30 days).

Path drift. The retrieval reports how many file paths cited in the body still exist on disk. If a memory says “see src/foo.py for the routing logic” and that file is gone, the missing-path count surfaces immediately.

Commit drift. When the caller is in a git repo that owns the memory, the retrieval reports how many commits have landed since the last verification. A memory verified three months and two hundred commits ago is fresh on the calendar but should be spot-checked anyway.

After a successful spot-check, the model calls memory_verify to refresh the timestamp and log a structured verified_paths attestation. If a claim drifted, the fix lands first via memory_update, then the model verifies against the corrected body.

Plain files on disk

No database. Every memory is a markdown file with YAML frontmatter in ~/.claude-memory/ (or ./.claude-memory/ for project-scoped, or wherever BETTERMEMORY_DIR points).

---
schema_version: 1
id: 01HXYZ123ABCDEFGHJKMNPQRST
created: 2026-03-14T10:23:00+00:00
updated: 2026-03-14T10:23:00+00:00
scopes: [tools, learning-style]
confidence: high
source: explicit-statement
---
When I ask for a "zero to hero" tutorial, I want a hands-on
walkthrough with code I can run, not a tour of the IDE.

It is greppable, diffable, hand-editable. To back up, copy a directory. schema_version is 1 and stays there. Every new feature reads and writes the same files.

What 2.0 added

Nine additive features, no on-disk breaking changes.

Claim-level provenance. memory_record_use(claim_excerpts=…) logs the load-bearing phrase the model actually leaned on for each memory. The event log can trace any response back to a specific sentence, not just a memory id.

Hybrid retrieval. memory_search(mode=…) accepts keyword, bm25, semantic, or hybrid. Hybrid runs Reciprocal Rank Fusion over keyword plus BM25, adding a sentence-transformers semantic leg when the embeddings extra is installed and the config opts in, degrading gracefully without either. Keyword was the default at launch; hybrid took over in 2.6.8.

Write-time groundedness gate. Opt-in via memory_write(groundedness_check=True, source_transcript=…). The server walks the body sentence by sentence and flags any sentence whose content tokens overlap the transcript by less than 30%. Returns {status: "ungrounded", claims: […]} instead of committing.

FTS5 inverted index. Files on disk stay canonical. The index is a derived cache at <store>/.index.sqlite, an FTS5 virtual table over body and scope text, kept live by Store hooks. memory_search uses it as a candidate pre-filter once the store crosses 500 memories. Falls back to a linear scan when the index is missing or corrupt; on-disk truth never blocks on the cache.

Git-based cross-host sync. Memory is markdown on disk, so git already handles the interesting cases: history, distributed copies, three-way merge. bettermemory sync init/status/push/pull/auto is a thin wrapper with a sensible .gitignore (excludes derived caches), post-pull index rebuild, and a no-op when nothing changed locally.

Local web UI. bettermemory ui runs a small FastAPI app on 127.0.0.1:8765: curation surfaces, searchable memory list, per-memory detail, one-click “Mark verified now”. Read-mostly. Writes happen in conversation. HTML rendered inline with html.escape, no template engine.

Consolidate, typed links, negative-outcome suppression. bettermemory consolidate is an offline pass: near-duplicate dedup, demote-never-applied, cold-scope and scope-typo suggestions. Dry-run by default. Typed inter-memory links (supersedes, contradicts, extends, depends_on) give graph-lite shape without a graph database, surfaced bidirectionally in memory_show. Every search hit carries recent_negative_outcomes when the memory was ignored or contradicted in the last 30 days and not since applied, so the model stops re-suggesting the same rejected memories.

Tombstones, not deletes

Removing a memory moves the file to .tombstones/ with removed and removed_reason appended. The body stays. memory_restore reverses a hasty removal; memory_write runs a tombstone-aware dedup pass, so a paraphrase of a removed memory months later gets flagged before re-introducing the wrong fact.

Confirmation tier for claims about you

Most memories (project facts, tooling references, infra notes) commit immediately on memory_write. Inferences about you are different. “The user prefers terse code review feedback” sticks if it lands wrong. So memory_write(category="user-inference") always returns status: "pending" regardless of config. The model asks in plain language, then calls memory_write_confirm.

Scoping by repo and worktree

Every memory written from inside a git checkout records the repo URL and worktree root in its origin block. memory_search filters hits to memories from the caller’s current repo and worktree, plus memories with no recorded origin (legacy entries or writes from outside any repo, which behave as global).

In the bettermemory repo you see bettermemory memories. Switch to the portfolio, those disappear. Sibling git worktree add checkouts stay isolated. Cross-project queries are explicit: pass auto_scope=false.

Past 2.0: the 3.x line

Everything above is the 2.0 surface. What came after is mostly additive (the on-disk format never broke), and it pulls in one direction: memory that holds up across agents and across time.

An episodes tier. Durable memory is for facts that should outlive the session. Loop iterations and subagent run-state (what we tried, what the takeaway was) are a different shape, and they had nowhere clean to live without polluting the durable store. The episode_* family (episode_write, episode_handoff, episode_search, episode_promote) keeps that journal under episodes/<session_id>/, TTL-pruned at 30 days and invisible to memory_search. A takeaway that turns out to matter gets episode_promoted into a real memory.

Multi-agent fan-in. The store was hardened for concurrent access and episodes gained a swarm_id, so a fan-out of agents can write run-state into one shared journal and a coordinator can read the whole swarm back; what one agent learns is there for the next.

An audit surface. It closes two retrieval-contract leaks that are otherwise invisible. memory_audit_turn re-runs the ranker over the turn that just finished and flags high-relevance memories the model didn’t retrieve. Silent misses. memory_health surfaces the inverse: memories the ranker keeps returning that the model never deliberately leans on, the search-result equivalent of a dead-letter queue. Both feed a curation rollup you can act on.

Write-reflex capture. Opt-in. The Stop hook queues durable statements the model forgot to save, surfaced through memory_proposals for review later, so a good fact mentioned in passing isn’t lost just because nobody said “remember that.” A companion auto-consolidate pass runs conservative dedup and demotion on an interval, with a richer consolidate --llm pass there on demand. It is maintenance the store does to itself.

A credential gate. The store is plain markdown, and sync pushes it across hosts over git, so a pasted secret would sit there unencrypted forever. memory_write now refuses secret-shaped tokens (vendor API keys, PEM headers, JWTs, guarded password= assignments) before they reach disk, and redacts the matched value from its own warning and the event log, so the gate never re-leaks what it refused.

Attested absence. Path drift had a false-positive class: a memory citing a remote host’s path, or a Linux config location while running on macOS, flagged as missing forever. memory_verify now takes verified_absent_paths, an attestation that a cited path is intentionally not on this machine, and the verdict stops nagging about it.

Curation that executes. memory_health diagnoses; memory_curate acts. It wraps the same consolidation engine the maintenance hook uses, dry-run by default, limited to reversible moves: tombstone near-duplicates, demote dead weight. And bettermemory try gives the staleness verdict a sixty-second offline demo: write a memory citing a file, delete the file, watch the next search flag it.

All of it lands as MCP tools or CLI subcommands: 25 tools now, 18 registered by default and most of the rest CLI-reachable when you want them. The 3.10 release was the unglamorous kind, roughly 150 heuristic defects found by multi-agent audit and fixed with a regression test each, the suite now at about 2,540. The verification trifecta (calendar, path, commit) is still the part I’d point at first. Everything since exists to make depending on it safe.

In practice

Day one. You say something durable.

When I ask for a tutorial, I want runnable code, not screenshots of an IDE.

Claude calls memory_write with category="user-inference" and scopes=["learning-style"]. The write goes pending. Claude asks: “Want me to remember that?” You confirm. A markdown file lands in ~/.claude-memory/.

Week two, fresh session:

Walk me through pandas from zero to hero.

That phrasing is ambiguous in a way stored preferences could resolve. Claude calls memory_search, surfaces the preference, and tells you up front: “Using your stored preference for code-driven tutorials…”

Month three:

What’s the difference between find and fd?

Generic. Claude doesn’t search. The reply isn’t tinted by months of accumulated personal context.

Install

For Claude Code:

/plugin marketplace add 0Mattias/bettermemory
/plugin install bettermemory@bettermemory

For any other MCP client (Claude Desktop, Cursor, Continue, Cline):

uv tool install bettermemory
bettermemory init --client claude-desktop

Idempotently merges the MCP server entry into the right config file. Restart the client and ask “what memory tools do you have?” If a batch of memory_ and episode_ tools shows up, you’re set.
